# EV-Dynamic-Dispatch 模块实现分析

本文档聚焦当前仓库 `ev_dispatch/` 目录下已实现模块，说明其职责、核心实现、模块间依赖关系，以及现阶段可改进点。

## 1. 模块总览

`ev_dispatch/` 当前采用分层结构：

- `core/`：领域模型与基础能力（位置、车辆、任务、充电站、路网、能量计算）。
- `algorithms/`：调度接口与具体策略实现。
- `simulator/`：动态仿真主循环，驱动任务生成与执行。
- `scenarios/`：场景构建器，负责初始化路网、车辆、充电站。
- `main.py`：演示入口，运行策略对比实验。

## 2. 各模块实现分析

### 2.1 顶层入口

#### `main.py`

**职责**
- 构建默认场景。
- 分别运行“最近优先”和“最大优先”策略。
- 输出完成任务数、失败数与总评分，进行策略对比。

**实现特点**
- 通过 `deepcopy(vehicles)` 保证策略对比公平，避免共享车辆状态互相污染。
- 新增了直接脚本运行兼容逻辑：当 `__package__` 为空时动态注入项目根目录到 `sys.path`，支持 `python ev_dispatch/main.py`。

**评价**
- 演示链路完整，适合作为课程项目 baseline。
- 目前输出以终端文本为主，缺少批量实验参数配置与结果持久化。

---

### 2.2 核心领域层 `core/`

#### `location.py`

**职责**
- 定义二维坐标对象 `Location`。
- 提供 `distance_to` 欧氏距离计算。

**实现特点**
- 采用 dataclass，结构清晰。
- 使用 `numpy.sqrt` 计算距离。

**评价**
- 作为统一地理抽象是合理的。
- 后续可扩展为真实路网坐标（如经纬度/路段节点映射）。

#### `vehicle.py`

**职责**
- 定义车辆状态（位置、电量、载重、效率、任务列表）。
- 提供可达性判断与剩余载重计算。

**实现特点**
- `can_reach` 使用距离 × 效率估算能耗，并预留保底电量 `reserve_energy`。
- `current_tasks` 当前仅作为状态字段，调度过程尚未形成完整任务队列执行逻辑。

**评价**
- 覆盖了作业核心约束（电量、载重）。
- 可进一步引入速度、服务时间窗、充电曲线等更细粒度属性。

#### `task.py`

**职责**
- 定义配送任务对象。
- 提供配送距离计算、超时判断。

**实现特点**
- 包含 `created_time`、`deadline`、`priority`，具备时效建模基础。
- `assigned_vehicles` 已预留多车协同可能。

**评价**
- 字段设计可支撑后续协同/优先级增强。
- 当前仿真中优先级和超时惩罚尚未充分进入调度决策主逻辑。

#### `charging.py`

**职责**
- 定义充电站资源状态（桩数、排队、功率）。
- 提供等待时间和充电时长估算。

**实现特点**
- 等待时间模型为简化近似：`max(0, (queue - num_chargers) * 30)`。

**评价**
- 对课程项目足够直观。
- 若追求高分，建议引入事件驱动队列或排队论模型（M/M/c 近似）。

#### `energy.py`

**职责**
- 统一能耗估算和充电站选择逻辑。

**实现特点**
- `calculate_consumption` 综合距离、载重系数、天气因子。
- `find_nearest_charging_station` 使用“距离 + 等待时间”的启发式目标。
- `required_distance` 参数目前保留未使用（以 `del` 消除警告）。

**评价**
- 已形成可扩展的能量管理入口。
- 需要注意：站点可达性判定使用“距离 <= 当前电量”，与实际能耗维度并不严格一致，后续可改为与效率模型一致的能耗约束。

#### `network.py`

**职责**
- 构建城市道路网络抽象。
- 暴露最短路接口（`dijkstra`、`a_star`）。

**实现特点**
- 当前用规则网格创建节点，并基于距离阈值连边。
- `dijkstra/a_star` 接口已给出，但当前实现退化为欧氏距离返回。

**评价**
- 接口设计对后续替换真实图算法友好。
- 现阶段尚未真正利用 `networkx` 的路径搜索能力，是主要可提升点之一。

### 2.3 算法层 `algorithms/`

#### `dispatcher.py`

**职责**
- 定义调度器抽象基类 `Dispatcher`。
- 约束策略实现统一签名：`assign_tasks(tasks, vehicles)`。

**评价**
- 抽象清晰，便于横向扩展策略。

#### `strategies.py`

包含两种贪心策略：

1. `DispatcherNearestFirst`
- 机制：在可用车辆与未分配任务中，迭代选择最近的可行 `(vehicle, task)`。
- 优点：实现简单、响应直观。
- 风险：易局部最优；车辆状态更新为“累加载重”，但没有任务完成后的卸货回收过程，长期仿真可能导致容量被持续占用。

2. `DispatcherLargestFirst`
- 机制：任务按重量降序，给每个任务选择最近可行车辆。
- 优点：可优先保障大任务。
- 风险：可能牺牲整体时效；同样存在载重状态回收不完整的问题。

**总体评价**
- 两策略已满足“多算法对比”的作业要求。
- 作为 baseline 很合适，后续可加入“收益密度”“deadline slack”“电量风险”等综合评分函数。

---

### 2.4 仿真层 `simulator/`

#### `simulator.py`

**职责**
- 按时间步生成随机任务。
- 调用调度器分配任务并模拟执行。
- 统计完成量、失败量、总里程、总评分。

**实现特点**
- 任务生成包含随机起终点、货重和截止时间。
- 完成奖励：`100 - dist * 0.1`；失败惩罚：`-50`。
- 低电量处理目前为简化逻辑：若电量 < 30，直接恢复满电。
- 返回指标包含平均任务得分（分母为 `completed + 1`，避免除零）。

**评价**
- 已形成完整仿真闭环。
- 仍有明显简化：
  - 充电过程没有真正调用充电站排队模型；
  - 任务是否超时未被纳入失败判定；
  - 车辆任务执行、装卸、时间推进比较粗粒度。

---

### 2.5 场景层 `scenarios/`

#### `default.py`

**职责**
- 提供默认实验场景构建函数 `build_default_scenario`。

**实现特点**
- 统一初始化路网、车辆（车场中心点出发）与充电站（按比例分布）。
- 参数化程度较好，可快速调整节点数、车辆数、站点数。

**评价**
- 适合演示和回归测试。
- 建议进一步拆分 `small/medium/large` 场景配置文件，便于固定实验基线。

## 3. 模块依赖关系

当前调用路径可概括为：

`main.py`
→ `scenarios.default.build_default_scenario`
→ `simulator.Simulator`
→ `algorithms.Dispatcher(具体策略)`
→ `core` 模型与工具（`Vehicle/Task/Location/EnergyManager/RoadNetwork`）

这一分层结构基本清晰：上层编排、策略决策、底层模型。

## 4. 与作业目标的匹配度

已满足的核心点：
- 动态任务生成与仿真循环。
- 车辆电量与载重约束。
- 两种调度策略对比。
- 充电站对象与排队时长估算接口。

尚待加强的点：
- 真实路径算法（当前最短路接口未落地图搜索）。
- 充电决策与队列机制在主仿真中的闭环整合。
- 时间窗、超时惩罚、优先级在调度目标中的显式建模。
- 多规模场景与批量实验统计输出。

## 5. 建议的下一步实现优先级

1. 路网层升级：让 `dijkstra/a_star` 真正基于图搜索并返回路径/代价。
2. 充电闭环接入：在仿真循环中调用 `EnergyManager.find_nearest_charging_station`，并使用 `ChargingStation` 队列状态推进。
3. 调度评分函数升级：将距离、deadline 松弛量、电量风险组合成统一代价函数。
4. 车辆状态一致性：引入任务完成后的卸货逻辑，避免 `current_load` 长期累积。
5. 实验工程化：增加固定随机种子、参数配置文件和结果导出（CSV/JSON）。

## 6. 结论

当前仓库实现已经具备“可运行、可对比、可扩展”的基础框架，结构清晰且符合课程项目初版要求。若按上述优先级迭代，可在较小改动成本下显著提升算法可信度、仿真真实性与报告说服力。
