# EV-Dynamic-Dispatch 项目结构与内容分析

## 1. 项目定位
本项目是一个面向新能源物流车队协同调度的大作业代码库，采用分层模块化设计。当前代码已形成可运行的最小闭环：
1. 构建默认场景
2. 按时间步动态生成任务
3. 调用调度策略输出动作
4. 在仿真中执行动作并计算结果
5. 输出文本化结果摘要

## 2. 顶层目录结构
当前仓库顶层目录如下：
1. README.md：项目说明与运行指南
2. requirements.txt：Python 依赖
3. docs：课程要求、架构文档、分工文档等
4. ev_dispatch：核心业务代码包

其中 docs 目录当前包含：
1. 2026-大作业要求.txt
2. ABC 对接文档.md
3. 分工.md
4. 项目结构与内容分析.md（本文档）

## 3. 核心代码结构分析

### 3.1 包总览
ev_dispatch 采用按职责分层的结构：
1. core：领域模型、路网、能耗、跨模块接口契约
2. algorithms：调度器抽象与具体策略
3. simulator：主仿真循环与动作执行
4. scenarios：场景构建
5. visualization：结果摘要展示辅助
6. main.py：演示入口

### 3.2 core 层
core 层承载模型定义和通用能力，属于系统底座。

1. location.py
- 定义二维坐标 Location
- 提供 distance_to 距离计算

2. vehicle.py
- 定义车辆状态：电量、载重、位置等
- 提供可达性判断 can_reach
- 提供剩余载重计算 get_available_capacity

3. task.py
- 定义任务对象：起终点、重量、时限、优先级
- 提供配送距离计算与超时判断

4. charging.py
- 定义充电站对象：充电枪数量、队列、充电功率
- 提供等待时间和充电时长估算

5. network.py
- 用 NetworkX 构建网格化路网图
- 提供 dijkstra 与 a_star 距离查询
- 当前路径查询返回最短距离，不返回完整路径节点序列

6. energy.py
- 提供能耗估算 calculate_consumption
- 提供充电站选择启发式 find_nearest_charging_station

7. interfaces.py
- 定义 A/B/C 协作关键契约：
  - Action：算法输出动作
  - SimulationState：仿真输入给算法的环境快照
  - SimulationFrame：仿真输出给可视化的帧数据
  - AbstractDispatcher：调度器抽象基类

### 3.3 algorithms 层
algorithms 层承载调度策略，当前以规则型贪心算法为主。

1. dispatcher.py
- 定义 Dispatcher 抽象类
- 约束所有策略实现 generate_actions(state)

2. strategies.py
- DispatcherNearestFirst：最近任务优先
- DispatcherLargestFirst：最大重量优先
- 两个策略均输出 Action 列表，动作类型目前主要是 assign_task
- 策略中已使用局部 planned_loads 避免规划阶段直接污染车辆真实状态

### 3.4 simulator 层
simulator.py 是当前系统运行中枢。

主要职责：
1. 随机生成动态任务（每个时间步）
2. 构造 SimulationState 并调用调度器
3. 执行动作并更新车辆、任务、得分
4. 生成 SimulationFrame 供可视化读取
5. 输出汇总指标（完成数、失败数、总里程、总评分）

关键特性：
1. 动作执行器 _execute_action 负责解释 Action 语义
2. 当前已支持 assign_task 动作
3. 每步结束会将低电量车辆直接补满电（简化充电逻辑）
4. 提供 get_frames 与 get_latest_frame 两个可视化接口

### 3.5 scenarios 层
scenarios/default.py 提供默认场景工厂函数。

内容包括：
1. 生成网格路网
2. 初始化车队（默认 5 辆）
3. 初始化充电站（默认 3 个）

当前状态：
1. 仅有 default 场景
2. 尚未形成 small/medium/large 多规模场景文件

### 3.6 visualization 层
visualization/console.py 当前提供轻量结果展示。

能力：
1. 输出完成任务数、失败任务数、总评分
2. 输出最后一帧的 step 与 pending 数量

定位：
1. 目前属于控制台汇总层
2. 已为后续图形界面或仪表盘扩展保留数据入口（SimulationFrame）

### 3.7 入口与运行流程
main.py 的工作流程：
1. 构建默认场景
2. 分别运行两种策略
3. 用 deepcopy 隔离车辆状态，保证对比公平
4. 输出策略对比结果

## 4. 当前数据流与调用链
按一次时间步来看，核心调用链如下：
1. Simulator 生成当步任务
2. Simulator 构造 SimulationState
3. Dispatcher.generate_actions(state) 生成 Action 列表
4. Simulator._execute_action 执行动作并更新状态
5. Simulator 生成 SimulationFrame
6. visualization 层读取 frame 进行展示

这是目前 A（算法）-B（仿真）-C（可视化）的标准协作主线。

## 5. 当前实现完成度评估

### 5.1 已具备能力
1. 模块化分层清晰
2. 两种调度策略可运行可对比
3. 动态任务生成已接通
4. 基础能耗模型已接通
5. A/B/C 对接接口已建立
6. 可视化消费仿真帧的通道已具备

### 5.2 仍为简化实现的部分
1. 动作语义较少，目前主要是 assign_task
2. 充电流程简化为低电量直接补满，未进入动作化排队充电
3. 路网最短路当前仅返回距离，不输出路径序列
4. 任务超时与优先级虽有字段，但评分中尚未完整体现
5. 场景规模目前仅默认场景，尚未完成多规模标准化

## 6. 建议的下一步迭代
1. 在 Action 中扩展 charge、idle、reposition 等动作，并在 _execute_action 完整实现。
2. 将充电决策与排队过程从“自动补电”改为“动作驱动+资源竞争”。
3. 在 scenarios 下新增 small、medium、large 场景，形成统一实验基线。
4. 在 network.py 增加路径节点序列返回接口，支持轨迹可视化。
5. 在评分模型纳入 deadline 与 priority，形成更符合课程要求的评价体系。
6. 在 visualization 层增加图表输出（趋势图、站点负载图、策略对比图）。

## 7. 结论
当前项目已从“单体脚本”演进为“可协作联调的模块化系统”，结构上已经具备 A/B/C 三方并行开发的基础。后续重点应从“可运行”提升到“约束真实、实验可比、展示完整”，以满足课程作业高分项要求。
